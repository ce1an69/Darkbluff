# 核心玩法

> 属于 DarkBluff 设计文档。主索引见 [design.md](design.md)；世界观设定见 design.md「世界观设定」。

## 游戏循环

```
进入案件/章节
    |
观察场景（表面世界）
    |
切换左眼（影子世界）获取对比信息
    |
通过 ask 指令收集更多线索
    |
分析表面/影子之间的矛盾与逻辑关系
    |
做出审判（judge，系统在审判前自动建立检查点）
    |
若完成本章必要审判 → 自动推进剧情
    |
进入下一章节 / 审判累积 → 走向不同结局分支
```

循环内的观察、切换、询问步骤可自由穿插，没有强制顺序；`judge` 后若本章 `required_judgments` 已全部完成，系统会自动推进到下一章或进入结局（见 [commands.md](commands.md)）。

> 审判本身没有选项、没有对错，但必要审判完成后会自动推进。若想探索其他走向，玩家可用 `map` 打开章节树/检查点地图，选择已经历过的 `chapter_start` 或 `before_judgment` checkpoint 回滚。

## 推理机制

- **表面世界信息 = 真命题**
- **影子世界信息 = 假命题**（取反即可得到真相）
- 玩家结合两侧的约束条件，缩小可能性范围，推导出真相

**真值模型是二元的、贯穿全游戏**：表面恒真、影子恒假。难度不通过"破坏二元真值"来提升，而是通过约束的数量、推理链的深度、跨场景线索的综合来提升（见下文「难度递进」）。

**示例**：

```
表面 NPC："我午饭吃了鸡蛋或苹果"（真）
影子 NPC："我吃了水果"（假 → 实际上没有吃水果）
推理：没吃水果 = 没吃苹果 → 吃的是鸡蛋
```

### 作者守则（影子文本）

影子文本必须是一条「其逻辑否定有意义且无歧义」的命题，否则玩家无法取反推理。创作约定：

- ✅ 好：「我吃了水果」→ 否定「没吃水果」干净明确
- ✅ 好：「我整晚都在酒馆」→ 否定「并非整晚都在酒馆」
- ⚠️ 避免：「我大概在家」「我好像见过他」——模糊量化词的否定歧义大，不应作为唯一约束
- ⚠️ 避免：影子文本与表面文本描述的是完全不同的事件（无法构成同一变量的约束）
- 一条影子文本应能与至少一条表面文本或其他影子文本（取反后）构成可解的约束组

引擎不强制校验「可取反性」（这是语义层面的约束），但 `darkbluff check` 会检查每个影子文本块是否有对应的表面对照，作为内容审校提示（见 [content-engine.md](content-engine.md)）。

## 难度递进

难度只沿以下轴提升，真值模型始终二元：

- **前期**: 简单直接取反（单角色、单信息、直接反转）
- **中期**: 多角色交叉引用，需要多步推理（A 的影子否定 + B 的表面肯定 → 缩小范围）
- **后期**: 约束数量增加、推理链变长、跨场景线索综合；部分命题需要收集更多事实才能消歧（"模糊"指**需要更多约束才能确定**，而非真值本身模糊）

## 审判的性质

`judge` 是玩家对某角色发起的**审判**：触发该角色一段固定的审判剧情，并记下「审判过该角色」。审判本身没有选项、没有对错——它是一次剧情演出：

- 推理（表面真 / 影子假）是为了让玩家**理解每个角色与事件**，从而审判要不要审判他
- 一章可以审判多个角色；**审判了哪些角色**经由自动推进时的章节跳转条件，导向不同的后续章节与结局（见 [narrative.md](narrative.md)「分支与结局」）
- 因此通过 `map` 回到审判前或章节开头、改审别的角色，是探索不同走向的方式——重玩与推理本身才是到达不同结局的方式
