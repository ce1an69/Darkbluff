# 剧情创作规则

> **每次写剧情内容前必须阅读本文档。** 机制层约束见 [gameplay.md](../gameplay.md) 和 [narrative.md](../narrative.md)。

## 文风准则

**剧情文本必须像写小说一样去写。**

你不是在填表，不是在写游戏教程，不是在给玩家发任务。你是在写一个发生在锈铁镇上的故事，这个故事恰好以游戏为媒介。每一段场景描述、每一行对话、每一次审判，都应该带给读者强烈的体验感——让他们闻到铁锈味，感到左眼切换时的不安，在某一句话面前停下来。

- 用具体的感官细节替代抽象概括。不要写"酒馆很热闹"，写"吧台上积了一层啤酒渍，角落里两个人在吵昨晚的事，第三个人趴在桌上，不知道是醉了还是死了。"
- 影子侧的文字要有重量。那不是"另一个版本"——是同一场景被剥掉了皮肤。
- 句子节奏要有长有短、能呼吸。不要频繁使用连续短句或大量句号拆碎段落；偶尔的短句可以制造节奏变化，但通篇碎片化会让读者喘不过气。用逗号和自然连接词把关联的内容串起来，长短交错。
- 对话是人物在活，不是信息在传输。让角色说只有他们会说的话。**对话为主，描写为辅**：让角色多说，让他们的台词承担叙事和揭示的大部分重量；动作描写、神态描写、氛围描写只保留对话发生的必要骨架，能砍就砍，不要写得比台词还多。
- 审判场景不是流程。是猫选择在谁的罪面前不再扭头。让这一刻有代价感。
- 记忆碎片是残影——画面、气味、触感、半句话。不要完整，不要干净。记忆碎开来才是记忆。
- 内在之声不给道德判断。它说"他在说谎"，不说"他是坏人"。
- **人称一致**：所有叙事文本（场景描述、intro、outro、心声、记忆碎片、走不出去、审判结果剧情、对话文件的 `>` 动作块）统一使用第二人称**"你"**称呼玩家。NPC 说出来的对话台词不受此限——NPC 可以自称"我"、可以称呼玩家为"你"。intro 中描述猫也使用"你"，不使用"他"或"猫"。
- **文件标题规则**：叙事触发器（心声/记忆碎片/旁白）的 Markdown 文件**不要**以 `# 标题` 开头——TUI 面板已经显示 `▌ 心声` / `▌ 记忆碎片` / `▌ 旁白` 标签，文件内再加同名 `#` 标题会造成重复。intro/outro/trial 等文件同理：若面板已通过其他方式给出标题，文件正文不应重复。

## 双世界规则

- 表面恒真、影子恒假。任何文本不可违反这条底线。
- 影子侧对话不是随机假话，是角色对其压抑真相的否认——他们说出来的，恰好是他们最希望自己相信的那个版本。
- 影子文本必须可否定且无歧义。避免模糊量化词（"大概""好像""也许"），避免与表面描述完全无关的独立事件。每一条影子文本都应该和至少一条表面文本构成可解的约束组。
- 取影子之反，结合表面之真，推理链成立。两侧约束共同缩小可能性。

## 推理验收规则

写任何章节前，先把本章的推理骨架列清楚；写完后按同一份清单自检。这里不要求把骨架展示给玩家，但作者必须能用它证明文本可解。

- 每章至少有一组核心约束：表面命题、影子命题、影子取反后的真命题、玩家能据此排除或确认的结论。
- 每个关键线索都要能追溯到数据：`clues.yaml` 的 `id`、`source`、`world`，以及对应对话文件里的 `## topic_id` 与 `[surface]` / `[shadow]` 文本。
- 每个会影响分支的话题或审判，都要能解释它为什么使用这个条件：`unlock_after` / `next.branches[].when` 引用了哪些线索、审判点或叙事触发器。
- 影子文本不能只为了气氛存在。若某段影子文本没有推理作用，也必须有角色塑造或主题作用；若两者都没有，删掉。
- 不把谜底写成结论句。玩家应通过两侧文本的冲突推出结论，而不是由角色、旁白或心声直接宣布。
- 不为校验器写文本。`darkbluff check` 只能保证引用完整，不能保证命题可取反；作者必须人工检查语义层面的可解性。

### 命题配对格式

剧情草稿或审校笔记中，每个关键推理点建议用下面格式记录，避免表面/影子关系失控：

```text
变量: 谁在何时何地做了什么
表面命题: ...
影子命题: ...
影子取反: ...
结论: ...
数据锚点: chapter / character.topic / world / clue_id
```

如果某个推理点无法填出「影子取反」或「结论」，说明它不能作为关键线索；要么补约束，要么降级为氛围文本。

## 主角规则

- 猫不主导对话。听，问，偶尔说一句。内在之声填补猫没说出口的那部分。
- 猫的动机是逃离。不是查案，不是正义，不是好奇心。写猫做任何选择时，问自己：他为什么觉得这能让自己离门更近一步。
- 猫的性格从一而终：警觉，独立，锐利。行动快于话语。不热血英雄，不冷血杀手。
- 内在之声不给道德判断。"他在说谎。""你看那边。"猫自己决定怎么回应。
- 记忆碎片全游戏共四次。不滥。每次只给残影——画面一闪，气味掠过，声音说了半句话。不给完整信息。

## 叙事触发器（心声 / 碎片 / 走不出去）

主角脑中的声音（心声）、闪回的残影（碎片）、走不出去的旁白——这些是**非对话叙事**，用章节的 `narrative` 字段承载（数据格式见 [../data-formats.md](../data-formats.md)「叙事触发器」）。它们不是 NPC 台词，是猫的主观层，在展示与 `note` 中独立归类。

**心声**:

- **纯对话形式**：心声文件只包含那个声音说的话。不要场景描写、不要动作叙述、不要猫的心理活动或身体反应。每一行都是那个声音在说话——像只有一个角色在念台词。
- 脑中另一个声音，对影子世界做独立评论。不给道德判断，只说「他在说谎」「你看那边」
- 不是旁白解说，不是心理独白流水账。是猫和（猫也分不清的）另一个意识之间的低语
- 可较频繁，但忌说教、忌替玩家下结论。声音不对猫做道德判断，猫自己决定怎么回应

**记忆碎片**:

- 全游戏 4 次核心碎片。一闪的残影——画面、气味、半句话、铁门关上的回声
- 不给完整信息，不给干净答案。记忆碎开来才是记忆
- 碎片的 `id` 同时是条件标记：可让「看过碎片后解锁某话题」（`unlock_after: 碎片id`）
- 每个碎片都必须有全局编号与用途：它补哪一块猫的过去、解锁或影响哪个话题/分支、为什么必须在这个章节出现。四次之外不得临时加“类碎片”绕开配额。

**走不出去**:

- 承载「猫无法离开锈镇」的世界规则，由边缘场景（镇边荒地）的 `exit_attempt` 承载
- 玩家**主动**尝试离开时触发——不是旁白告知，是玩家自己试、自己发现走不出去
- 首次完整展示（一段叙事），之后简短。不滥用，保留首次的冲击

**共同文风**:

- 这些文本是小说的一部分，不是系统提示。用感官细节、用残影、用猫的主观
- 心声 / 碎片 / 旁白在展示与 note 中独立归类，写作时心里有这个区分——它们和 NPC 对话质地不同

## 角色规则

- 善恶不存在。没有纯黑纯白的角色。每个人都有苦衷，每个人也都干过脏事。写一个"坏人"的时候，先想他回家之后会做什么。
- 角色有独立于主线的生计、琐事、癖好。酒馆老板不只是"信息源"——他擦杯子有固定的手势，他记得每个熟客第一杯喝什么。
- 大量对话应和线索无关。是人物在活，不是证人在供述。
- 精神畸变者不是"超能力者"。没有分级，没有命名，没有修炼体系。只是一种心理特质差异——像有人天生擅长撒谎，有人天生不擅长藏情绪。

## 风格规则

- 荒诞，天马行空。对话质地可以突然转弯。血腥下一秒可以是冷笑话，日常下一秒翻出深渊。永远不让读者猜到下一段的温度。
- 血腥暴力真实。不是耍帅，不是氛围滤镜。是脏的，快的，有后续的。打完架的人第二天走路还瘸。
- 黑暗中有温馨。食物是热的，劳动是实的，一杯酒可以只是酒，也可以是一句没说出口的"我懂"。
- 幽默来自角色的私人怪癖，不来自"搞笑段子"。没有人讲笑话。但有人说话让你想笑。
- 不让全知叙述者解释世界。没有人说"这是因为"。存在就是存在。

## 七宗罪结构

- 七章主线，七宗罪。章程序号等于剥开的深度。
- 第 1–2 章，傲慢与嫉妒：猫是远距离观察者。罪在他人，猫还干净。
- 第 3–5 章，暴怒、懒惰与贪婪：暴力与体制。猫被卷入，身上开始沾血和铁锈。
- 第 6–7 章，暴食与色欲：被告是猫的镜像。猫在审判别人时看见自己。
- 第 8 章，自我：猫成为自己的被告。四条世界线，四种自审。
- 每场审判不是"惩恶"——是猫选择在谁的罪面前不再扭头。

## 结构规则

- 主干七章，加四条世界线终章。单次游玩八章。
- 每章 15–30 分钟可玩内容。
- 审判无选项、无对错。审判本身是固定剧情。"审了谁"决定走向。
- 所有世界线平等。不在单线内暗示"这条不是真结局"。
- 终章 `ending: true`，完成必要审判后进入结局界面。
- 每一层剥开后，前期场所和对话将被玩家重新理解。写前期内容时，要为后期的重新解释留好空间——让同一句话在第二章和第七章读起来像两句话。

## 数据落地规则

剧情内容可以晚写，但规则和数据锚点必须先想清楚。新增章节草稿至少要说明以下内容，才能进入 YAML/Markdown 编写：

- 章节 `id`、标题、七宗罪位置、起始场景、涉及角色。
- 本章可审判角色列表，以及每个 `judgment_id` 对应的叙事代价；审判剧情不是“判对/判错”，而是猫选择看见谁。
- 可问话题分层：默认可问、由线索解锁、永久不可问。永久不可问的话题必须有叙事理由，不得作为 TODO 占位。
- 分支条件：哪些 `clue_id` / `judgment_id` / `narrative_id` 会进入 `next.branches[].when`，默认分支代表什么走向。
- 叙事触发器：心声、碎片、走不出去分别使用什么 `id`，触发条件是什么，是否进入 `note` 回顾。
- 场景覆盖：如果章节覆盖全局场景描述，surface 和 shadow 必须同时考虑；不能只写一侧。

## 审校清单

提交剧情数据前，按以下顺序检查：

1. 先跑 `darkbluff check`，修完所有 error；warning / advice 需要逐条确认，不应机械忽略。
2. 人工检查每个关键影子文本是否可取反、取反后是否与表面文本构成约束组。
3. 从玩家视角走一遍：只读已经出现的文本，能否理解当前可做的事，但不会被直接告知答案。
4. 检查 `note` 价值：玩家回看对话、叙事、审判时，是否能重新推理，而不是只看到流程记录。
5. 检查分支公平性：每条世界线都自洽，不在单线内暗示“这里不是真结局”。
6. 检查文风：删掉教程腔、设定解释腔、作者替角色说话的句子。

## 禁止

- 引入物理超自然体系外要素（怪物、魔法、神明）。
- 打破或绕开表面真/影子假的二元模型。
- 猫变热血英雄或冷血杀手。
- 把章节写成侦探查案模式。
- 影子侧"谎话"机械取反——每句都是精确的逻辑否定。
- 单线内暗示"这条不是真结局"。
- 写"游戏文本"。写小说。
- 用旁白、心声或角色台词解释数据规则、操作提示、分支条件。
- 为了凑线索写无人格的信息问答。
