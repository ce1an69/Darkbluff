# 叙事结构与流程

> 属于 DarkBluff 设计文档。主索引见 [design.md](design.md)。章节跳转的数据定义见 [data-formats.md](data-formats.md)「章节元数据」；检查点/回滚的实现细节见 [save-system.md](save-system.md)。

## 章节系统

- 每个章节对应七宗罪之一，剥开一层（约 15-30 分钟）
- 七章构成连续主线；终章（第 8 章）为猫审判自己，不同世界线导向不同的自审结果
- 不存在「唯一真相」——不同世界线包含不同的线索与视角碎片。玩家经历的线越多，对这座镇子和猫的了解越多，最后依靠想象力拼出每个人心中的那一种真实
- **首章 = 章节树的唯一根节点**（无任何章节的 `next` 指向它）；新游戏从首章开始，引擎启动校验有且仅有一个根（见 [content-engine.md](content-engine.md)）
- 进入任何章节时（新游戏首章 / 自动推进跳转 / `map` checkpoint 回滚），若该章定义了 `intro` 开场文本，先展示开场，玩家确认后再进入 `starting_scene`；省略则直接进入场景

## 分支与结局

- `judge` **审判没有对错之分**：审判没有选项，审判了哪些角色累积影响章节跳转，导向不同的世界线
- **世界线模型**：不同审判序列坍缩到不同世界线。每条世界线内部自洽、完整、合法，揭示的是那条线内的真实。没有哪条线是「真结局」——所有世界线平等。玩家经历多条线后，用想象力拼出属于自己的那一种真实
- 终章在 `chapter.yaml` 中标记 `ending: true`，没有 `next` 字段
- 进入终章后仍可探索场景、对话和审判；完成终章最后一个必要审判时记为达成该结局，展示 `outro`（若有）后进入结局界面
- 鼓励重玩以探索不同结局；已达成结局记录在 append-only 的 `discovered.endings` 中（set 去重），任何回滚都不丢失
- **终章的两阶段生命周期**: 通过前一章自动推进**进入**终章（此时只记 `discovered.chapters`），在终章中完成最后一个必要审判时才记入 `discovered.endings`、展示 `outro`（若有）并进入结局界面
- **探索进度反馈**：标题界面显示「已发现结局 X / Y」（X = `discovered.endings` 去重后数量，Y = 内容中 `ending: true` 的章节总数），给玩家探索目标感

## 终章结构

终章（`ending: true`）与其他章节使用相同的数据结构，但行为有约束：

- 终章**没有 `next` 字段**：完成最后一个必要审判时不评估跳转条件，而是将本章记入 `discovered.endings`、展示 `outro`（若有）并进入结局界面
- 终章可以包含场景、对话、审判，让玩家在结局中仍有探索与收束的体验；其 `required_judgments` 需先完成才能结束
- v1 不支持无审判结束章节；如果需要纯叙事收束，应通过一个叙事型审判点承载章节结束动作
- 每条世界线的终章揭示该线内的故事收束。各线线索相互补充但不依赖——每条线是独立的完整故事，多线叠加后玩家对世界的理解更深，但不存在唯一的「正确拼法」
- 终章可定义 `outro`（可选）：与 `intro` 对称，`intro` 是进入章节时的开场叙事，`outro` 是完成终章最后一个必要审判后的结局收尾叙事。`outro` 仅对终章有效，非终章定义为启动校验错误

## 结局界面

完成终章最后一个必要审判后（展示 `outro` 完毕或无 `outro`），进入结局界面（`Ending` 状态），展示：

- 终章 `title` 作为结局名（如「幸存者」「参与者」等）
- 「已发现结局 X / Y」探索进度
- 菜单选项：**返回标题** → 回到标题界面

## 重玩与检查点

游戏采用单存档自动保存，不提供多个手动存档槽位。探索不同分支通过游戏内 `map` 指令完成：

- 系统在**章节开始**和**每次审判前**自动创建检查点（见 [save-system.md](save-system.md)）
- `map` 展示已经历过的 `chapter_start` 与 `before_judgment` checkpoint
- 选择 `before_judgment` = 回到某次审判前，便于改审别的角色
- 选择 `chapter_start` = 回到某章开头，便于重走整章
- 选择早前章节的 checkpoint = 销毁性跨章回滚（**需二次确认，Esc 可取消**），丢弃回滚点之后的当前流程进度与检查点
- `discovered` 保留——已体验的章节/结局/问过的话题永久记忆，不因回滚丢失
- 回滚到章节开头后从该章 `intro`（若有）重新开始，玩家可做出不同审判，导向其他分支

> 检查点不保存完整状态快照，只记录位置 + 当前章节数组的长度（见 [save-system.md](save-system.md)）。所有探索记忆（到过的章节/结局/选过的审判）记录在 append-only 的 `discovered` 中，任何回滚都不截断它。

> **已知权衡**：单存档下，`map` checkpoint 回滚是销毁性的——会丢失回滚点之后的当前流程进度（但 `discovered` 保留分支记忆）。这是单存档设计的取舍；若未来需要非破坏性探索，可引入独立的「重玩槽」，但会突破单存档约束。

## 启动与标题界面

**首次运行**（无存档）：直接进入标题界面，仅「新游戏」可选。

**标题界面菜单**：

| 入口   | 行为                                                         |
| ------ | ------------------------------------------------------------ |
| 新游戏 | 若已有存档，二次确认覆盖（单存档）；初始化空存档并进入第一章 |
| 继续   | 加载存档，恢复到 `current_*` 位置                            |
| 退出   | 退出程序                                                     |

标题界面同时显示「已发现结局 X / Y」作为探索进度反馈。

**设置持久化**: 底层存档系统支持独立的 `settings.json`（与存档同目录），不混入 `save.json`。当前标题菜单尚未提供「设置」入口；`--no-motion` 为后续 TUI 实装后的临时覆盖参数。
